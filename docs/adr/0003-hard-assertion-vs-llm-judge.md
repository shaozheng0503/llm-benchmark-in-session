# ADR-0003: 硬断言 vs LLM-as-Judge 的边界

## 状态

2026-06-01 已采纳

## 背景

评分有两条路：
- **硬断言**（regex / JSON / 长度）：快、便宜、可复现，但覆盖不了开放题
- **LLM-as-Judge**：可处理开放题，但慢、贵、有偏差

## 决策

**硬断言为主，LLM-as-Judge 为辅**：

| 题型 | 用什么 |
|------|--------|
| 身份、JSON 抽取、关键词命中 | 硬断言 |
| 风格迁移、代码质量、自由发挥 | LLM-as-Judge |
| 评分员间一致性 / 自身有偏 | 双裁判 + Cohen's Kappa |

`judge.py` 同时支持单裁判 / 双裁判 / 双裁判+仲裁三档。

## 已知 LLM 偏差

通过 `judge_bias.py` 检测：
1. 位置偏好（A vs B）
2. 长度偏好（长答案得分虚高）
3. 自我偏好（同源模型评同源输出偏高）

建议：rubric 显式"忽略长度差异" + 匿名化答案 + 随机化位置。

## 退出条件

如未来引入强模型作为主裁判，可考虑：
- AutoCritic（自动 rubric 优化）
- Pairwise ranking（避免绝对分数的偏差）
